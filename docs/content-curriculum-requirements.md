## 1. Purpose

This document defines the content structure, curriculum units, metadata, and content packaging requirements for AudioVerse English.

## 2. Scope

The scope includes the structural definition of lessons, units, and levels, the requirements for transcripts, and the structure and versioning of downloadable offline content packages. This document does not define the actual English pedagogical content, but rather the software requirements for supporting that content.

## 3. Audience

- Content creators and instructional designers
- Software engineers implementing the data and content layers
- Quality assurance for testing content package ingestion

## 4. Definitions

See [Glossary](glossary.md) for definitions of Lesson, Course, Unit, Soundscape, and Content Package.

## 5. Assumptions

- Content is authored outside of the application and bundled into versioned packages.
- The application will ingest these packages locally from the filesystem.

## 6. Requirements

### 6.1 Curriculum Hierarchy (CONT-001)

The curriculum must be structured hierarchically:
- **Course/Level:** The highest level of categorization (e.g., "Beginner", "Intermediate").
- **Unit:** A thematic grouping within a Course (e.g., "At the Airport", "Greetings").
- **Lesson:** The atomic unit of instruction, which contains audio instructions, practice attempts, and soundscapes.

### 6.2 Lesson Structure (CONT-002)

Each Lesson must support, at minimum, the following components:
- **Metadata:** Title, description, difficulty level, estimated duration, and accessibility tags.
- **Audio Prompts:** The pre-recorded audio instructions and explanations.
- **Target Phrases:** The specific English phrases the learner is expected to practice and be evaluated on.
- **Transcripts:** Text representations of all spoken audio prompts, vital for Low Vision users and useful for testing.

### 6.3 Content Packaging (CONT-003)

Content must be distributed in discrete "Content Packages" (e.g., ZIP files containing JSON metadata and media assets).
- A Content Package must be entirely self-contained, meaning it requires no external network requests to function once downloaded or pre-installed.
- Each Content Package must include a manifest file (e.g., `manifest.json`) defining the curriculum hierarchy, metadata, and relative paths to all associated audio assets.

### 6.4 Versioning (CONT-004)

Content Packages must be versioned.
- The application must support ingesting newer versions of a Content Package.
- When a package is updated, the user's progress against previously existing lessons within that package must not be lost.

### 6.5 Soundscape Manifests (CONT-005)

Soundscapes (defined fully in Phase 14) are a specific type of lesson content.
- Soundscapes must be defined by a JSON manifest detailing the audio layers, timing, and spatial metadata.
- Soundscapes are strictly asset-driven; they are not dynamically generated by AI at runtime.

## 7. Risks

- **Content Size:** High-quality audio files can make Content Packages very large, violating the low-end device storage constraints. *Mitigation: Define strict compression standards and allow granular downloading/deletion of individual Units.*
- **Version Migrations:** Updating content might break user progress if IDs change. *Mitigation: Content IDs (for Courses, Units, Lessons) must be immutable UUIDs.*

## 8. Open Questions

- What specific compression format and bitrate should be mandated for audio assets to balance quality and file size?
- Should we allow downloading individual Lessons, or only entire Units/Courses? (Recommendation: Download at the Unit level to simplify dependency management).

## 9. Acceptance Criteria

- Content architecture allows representation of the full hierarchy (Course -> Unit -> Lesson).
- A sample JSON manifest correctly describes a lesson and its assets.
