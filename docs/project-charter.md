## Purpose

The purpose of this document is to define the core mission, scope, stakeholders, success criteria, constraints, and explicit non-goals for AudioVerse English. This charter establishes the foundational boundaries and direction for all subsequent engineering, design, and product decisions.

## Problem Statement

Blind and low-vision learners face significant barriers when using traditional English learning applications, which heavily rely on visual cues, drag-and-drop mechanics, and complex visual navigation. Additionally, many learners reside in areas with unstable internet access or rely on low-to-mid-range mobile devices, making cloud-dependent or visually heavy applications unusable.

## Mission

To provide an accessible, offline-first, audio-based English learning platform designed specifically for blind and low-vision learners, enabling independent language learning through structured lessons, audio soundscapes, and speaking practice.

## Scope

The primary scope of the AudioVerse English project includes the development of a mobile application (starting with an MVP) focusing on the core learning journey for blind and low-vision users. This includes:

- Accessible onboarding tailored for screen readers (TalkBack, VoiceOver) and low-vision needs.
- Offline access to lesson content and progression tracking.
- Audio-first lesson playback and interaction.
- Speaking practice functionality utilizing pluggable speech recognition and pronunciation assessment.
- Optional features such as an AI Tutor and cloud synchronization that can be gracefully degraded or completely disabled without compromising the core offline learning experience.

## Stakeholders

- **Primary Users:** Total Blind and Low Vision learners.
- **Secondary Users:** Caregivers or teachers assisting the primary users (though independent use is the goal).
- **Engineering Team:** Jules (AI Agent) and human maintainers/reviewers.
- **Product Owners:** Visionary project leads ensuring adherence to accessibility and educational goals.

## Success Criteria (Goals)

For the MVP and early releases, success is defined by:

1. **Accessibility Verification:** A Total Blind learner can successfully complete onboarding, navigate the app, and finish a core lesson entirely without visual assistance (using screen readers, haptics, and audio cues).
2. **Offline Reliability:** The application must function meaningfully without an internet connection, allowing for lesson completion and progress tracking.
3. **Performance Baseline:** Acceptable performance metrics (startup time, latency, battery use) on low-to-mid-range target devices.
4. **Traceability:** Complete and approved documentation mapping features back to foundational requirements.

## Constraints

- **Accessibility First:** Visual aesthetics must never override accessibility needs (e.g., contrast ratios, text scaling, screen reader focus order).
- **Device Limitations:** Must operate efficiently on low-to-mid-range smartphones.
- **Modular Engines:** Core technologies like Speech Recognition (ASR), Pronunciation Scoring, and AI Tutors must be implemented as modular, pluggable components to allow future replacements without systemic rewrites.
- **Privacy and Data:** Voice recordings and user progression must default to local-first storage. Any data leaving the device must require explicit consent.
- **Documentation Gate:** No application code can be written until explicit documentation gates (as defined in the Handbook) are passed.

## Non-Goals

To maintain focus and adhere to the project constraints, the following are explicitly identified as non-goals for the initial phases and MVP:

- **Complex Social Features:** Leaderboards, public profiles, and heavy social interaction.
- **Mandatory Cloud Dependency:** Any feature that requires a continuous internet connection to complete core learning tasks.
- **Generative Soundscape Creation:** Soundscapes must be structured, pre-defined audio assets (JSON + WAV/MP3), not dynamically generated by AI on the fly.
- **Heavy Embedded AI (without benchmarks):** Including large language models or AI components that have not passed strict performance and battery benchmarks on low-end devices.
- **Advanced Gamification:** Intricate visual gamification that distracts from the core audio learning experience or degrades accessibility.
